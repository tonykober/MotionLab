CrowdSim3D — 深度調研報告

工具概述

CrowdSim3D 是 Blender 專用的代理式群眾與交通模擬外掛，由荷蘭開發者 Monique Dewanchand（公司 @tmind）開發。它透過 Blender 節點編輯器讓使用者以視覺化方式編排群眾行為，支援路徑跟隨、障礙迴避、狀態機、戰鬥模擬、交通號誌等功能。

解決的問題：

  • 手動擺放和動畫化大量角色極度耗時
  • Blender 原生缺乏真正的代理式群眾模擬系統
  • 戰鬥場景中需要角色自主決策、互動和碰撞迴避
  • 交通模擬需要號誌、車輛路線、行人協調

功能詳細說明

核心架構

CrowdSim3D 是代理式（Agent-Based）模擬器，每個角色是獨立的「代理」，具有自己的行為邏輯。CGDive 評價其為「Blender 生態中最進階的代理式模擬器」。

主要功能

功能說明
節點式行為編輯使用 Blender Node Editor 視覺化編排群眾邏輯
導航演算法支援多種路徑規劃和導航算法
障礙迴避力場式迴避（Force Avoidance），群眾間和障礙物迴避
速度式動畫混合根據移動速度自動混合 walk/run 動畫
狀態機（States）代理可切換不同狀態（如歡呼→墨西哥波浪→靜止）
路徑跟隨群眾沿指定路徑移動
目標導航群眾自動導向目標位置
陣型（Formations）軍隊式陣型擺放和移動
戰鬥模擬多個群眾混戰、個體決策、互動行為
交通模擬路口號誌、車輛路線、公共運輸、行人
網格/材質變化自動變換角色的 mesh 和材質，避免重複感
動畫排程（Animation Scheduling）控制動畫觸發時機和混合
模型特定變數每個代理可有獨立變數影響行為
回放功能（Playback）模擬結果可回放

群眾行為的複雜度層級

根據 Blender Conference 2023 演講「Interacting Crowds」：

  • 基礎：群眾整體移動（行軍）
  • 進階：個體決策 + 群體行為混合（戰鬥場景）
  • 互動：角色之間相互反應、狀態切換

交通模擬

  • 路口有/無號誌配置
  • 車輛路線設定
  • 公共運輸路線
  • 行人模擬
  • 車輛方向燈動畫

安裝/使用流程

安裝

  1. 從 CrowdSim3D 官方網站購買/下載（有免費小規模版本）
  2. Blender → Edit → Preferences → Add-ons → Install → 選擇 zip
  3. 啟用 CrowdSim3D
  4. 購買版包含範例檔案（含角色和動畫，CC-BY-NC 授權）

基本使用流程


1. 建立角色模型（或使用範例角色）
2. 設定角色 mesh/材質變化（增加多樣性）
3. 建立群眾群組（Crowd Group）
4. 設定陣型/放置方式
5. 在 Node Editor 中編排行為邏輯：
   - 導航目標
   - 迴避規則
   - 狀態切換條件
   - 動畫映射
6. 設定環境（障礙物、路徑、號誌）
7. 執行模擬
8. 回放/渲染

優缺點分析

優點

  • ✅ 真正的代理式模擬：每個角色獨立決策，不只是粒子系統
  • ✅ 節點式編輯：視覺化程式設計行為，無需寫程式碼
  • ✅ 戰鬥場景支援：陣型、互動、個體決策——遊戲動作需求核心
  • ✅ 速度式動畫混合：自動在 walk/run 之間混合，結果更自然
  • ✅ 交通模擬：額外附帶城市/交通場景能力
  • ✅ Blender 原生整合：直接使用 Blender 的 Node Editor
  • ✅ 持續開發：多次 Blender Conference 演講，積極維護
  • ✅ 有免費版本：可先試用小規模群眾

缺點

  • ❌ 價格較高：€150-200 對獨立開發者而言不便宜
  • ❌ 效能限制：大量代理可能影響效能（純 Python/Blender 內計算）
  • ❌ 文件偏少：主要依靠範例學習，官方文件不如 ARP 完善
  • ❌ 不直接匯出引擎：模擬結果在 Blender 中，需另行匯出
  • ❌ 角色需自備或使用範例：不含大量現成角色庫
  • ❌ Blender 版本依賴：早期版本（2.79 起）到現在的相容性需注意

與現有工具的整合可能性

與 Kimodo 的整合

環節整合方式
Kimodo 動作 → CrowdSim3D 動畫庫用 Kimodo 批次生成 walk/run/idle 等基礎動作，作為 CrowdSim3D 的動畫資產
AI 生成多樣性用 Kimodo 為同一動作生成多個變體（不同速度/風格），增加群眾多樣性
戰鬥動作Kimodo 生成攻擊/防禦/倒地動作，CrowdSim3D 在狀態機中觸發

與 Blender → UE/Unity Pipeline 的整合

流程說明
CrowdSim3D 模擬 → 烘焙 → FBX 匯出將群眾動畫烘焙為骨架動畫後，用 ARP 或原生 FBX 匯出
引擎內群眾 vs Blender 預渲染若為過場動畫/CG，可在 Blender 中完成全部；若為即時群眾，需引擎自己的系統
陣型/路徑規劃預視在 CrowdSim3D 中預視群眾行為，確定邏輯後在引擎中實作

建議整合流程


Kimodo 批次生成動作庫（walk/run/attack/idle...）
→ CrowdSim3D 設定群眾行為（狀態機 + 動畫映射）
→ 模擬結果烘焙 → ARP 匯出 FBX → 遊戲引擎（過場動畫用）

對「AI 動作量產工作流」的貢獻定位

定位：AI 動作的群眾級消費者 + 行為編排引擎


[AI 動作生成] → [動作庫] → [CrowdSim3D 行為編排] → [群眾場景]
  (Kimodo)      (walk/run/   (狀態機+路徑+         (數百角色同場景)
                 attack...)    陣型+迴避)

具體貢獻：

  1. 動作消費放大：一個 Kimodo 動作可被數百個代理同時使用，一個動作的價值被放大 100 倍
  2. 多樣性需求驅動：群眾場景需要大量動作變體，驅動 AI 生成更多樣化的動作
  3. 戰鬥場景自動化：設定好行為邏輯後，戰鬥動畫由模擬器自動編排，減少手動 K 幀
  4. 場景預視：在投入引擎實作前，用 CrowdSim3D 快速驗證群眾行為設計
  5. 過場動畫量產：遊戲中的過場群眾動畫可直接在 Blender 中完成

限制：

  • 不是即時引擎解決方案（UE/Unity 有各自的群眾系統）
  • 最適合預渲染/過場動畫/概念驗證
  • 大規模場景（1000+ 代理）的效能需測試
