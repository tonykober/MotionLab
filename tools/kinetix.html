Kinetix — 深度調研報告

七、Pipeline 整合分析

項目說明
輸入來源影片 / AI 生成 / 動作庫
輸出格式FBX、GLB（遊戲引擎相容）
骨架標準遊戲 Humanoid 標準
上游工具影片拍攝 / Kinetix 動作庫
下游工具Unity（SDK）、UE5、Web3 遊戲引擎

推薦搭配：Kinetix SDK → Unity/UE5 即時角色 Emote 系統。專為遊戲角色 Emote/表情動作設計，適合需要 UGC 動作系統的多人遊戲。

對玩家端（使用流程）

  1. 在支援 Kinetix 的遊戲中找到「創建動作」入口
  2. 選擇輸入方式：上傳影片 / 輸入文字描述
  3. AI 雲端處理生成動畫
  4. 在遊戲內預覽效果
  5. 確認後套用到角色

注意：此為 B2B 工具，個人動畫師/開發者無法直接當作動捕工具使用。

  • 遊戲中使用的 AI 生成動畫歸遊戲開發者/玩家所有
  • 按 API 呼叫量計費（類似 SaaS 模式）
  • 需簽訂商業合約

6. 優缺點分析

優點

  • ✅ 遊戲內原生整合：玩家無需離開遊戲即可創建動畫
  • ✅ Text-to-Animation：文字描述直接生成，降低創作門檻
  • ✅ 多引擎支援：Unity + Unreal + Web
  • ✅ UGC 生態：玩家創建的動畫可交易，增加遊戲生態
  • ✅ 雲端處理：無需客戶端算力
  • ✅ 內容審核：避免不當內容

缺點

  • ❌ B2B Only：個人動畫師無法直接使用
  • ❌ 黑盒子：AI 模型細節不透明
  • ❌ 依賴雲端：無離線模式，需持續付費
  • ❌ 品質不確定：AI 生成品質可能參差不齊
  • ❌ 價格不透明：需個案洽談
  • ❌ 非 BVH 工具：不是傳統動捕/動畫製作工具
  • ❌ 玩家動畫品質受限：非專業人士上傳的影片品質可能不佳

推薦整合方式

Kinetix 和 Kimodo 不是同一賽道：

  • Kinetix = 遊戲內玩家 UGC
  • Kimodo = 開發期間的動畫量產

開發階段：[Kimodo] → [BVH] → [ARP] → [引擎]（角色預設動作庫）
營運階段：[Kinetix SDK] → [遊戲內玩家 UGC 動畫]（玩家自訂 emotes）

兩者可在同一遊戲中共存：Kimodo 做基礎動作庫，Kinetix 做玩家 UGC 層。

8. 對 AI 動作量產工作流的貢獻定位

定位：遊戲營運期的玩家 UGC 動畫平台（非開發期工具）


┌─── 開發期（我們的工作）──────────────────┐
│ [Kimodo] → [BVH] → [ARP] → [引擎動作庫]  │
└────────────────────────────────────────────┘

┌─── 營運期（遊戲上線後）──────────────────┐
│ [Kinetix SDK] → [玩家 UGC 動畫生態]       │
└────────────────────────────────────────────┘

量產流程中的角色：

  • 不在「動作設計師」的核心工作流中
  • 是給「遊戲設計師/製作人」的營運工具建議
  • 如果我們的遊戲需要玩家 UGC 動畫功能，Kinetix 是考慮選項
